同样用 GPT-5.4，我觉得 Codex 比 OpenCode 更省 token

如果底层都用 GPT-5.4，我的体感是 Codex 比 OpenCode 更省 token 和使用量。核心不是模型不同，而是 compaction 被公开说明到什么程度。

Mar 19, 2026
阅读时长: 3 分钟
语速

同样都是用 GPT-5.4，为什么我会觉得 Codex 比 OpenCode 更省 token，也更省使用量？

我的结论很直接：差别不在模型本身，而在 compaction 这件事到底被公开说明到了什么程度。

先说明一下，这篇文章不是严格 benchmark，也不是我拿一堆实验数据硬跑出来的结论。它更像是我根据公开文档，再结合自己实际用下来的体感，做的一个判断。

先说结论

如果你讨论的是 GPT-5.4 本身，那么模型没有变。

但如果你讨论的是“通过 Codex 用 GPT-5.4”和“通过 OpenCode 用 GPT-5.4”，那我的体感就是：Codex 往往更省。

我现在越来越觉得，这里面最关键的点，不是两边都叫 compaction，而是 Codex 公开讲清楚了 server-side compaction，OpenCode 没有。

关键不是 compaction，而是 server-side compaction

如果说的是 Codex，尤其是 OpenAI 文档里那套 server-side compaction 逻辑，那关键点其实非常明确：它说的不是一个泛泛的“压缩上下文”功能，而是 server-side compaction。

这件事为什么重要？因为它意味着上下文压缩不是用户自己在本地想办法整理历史，也不是一个名字上看起来差不多的配置项，而是服务端在上下文达到阈值之后，直接在链路里触发 compaction，并把 compacted context 带到后续轮次里继续跑。

OpenAI 文档还明确写了，compaction item 会用更少的 tokens 携带之前的关键状态。那这个意思其实已经很清楚了：后续请求不需要一直原样背着完整旧历史走。这样一来，后面的输入更短，token 和延迟自然就更容易往下掉。

所以如果我只看公开资料，那我会觉得 Codex 这里的逻辑是闭合的。它不只是说“我有 compaction”，而是把 compaction 怎么进入后续请求链路这件事说清楚了。

OpenCode 也有 compaction，但它不是 server-side compaction

OpenCode 本身就有 compaction，这点没什么争议。OpenCode 配置文档 里也直接给了相关配置。

一个常见配置大概长这样：

{
  "compaction": {
    "auto": true,
    "prune": true,
    "reserved": 10000
  }
}

这几个选项都不复杂：

  • auto：上下文快满的时候自动压缩
  • prune：优先移除旧工具输出，减少噪音
  • reserved：预留一部分上下文窗口，避免压缩时顶满

但我后来去看了源码，发现 OpenCode 这套 compaction 更准确的理解，不是 server-side compaction，而是它自己在应用层做的一套会话压缩。

大致流程是这样的：

  • 上下文快满时触发 compaction
  • OpenCode 调一个隐藏的 compaction agent 去生成 summary
  • 后续请求不再继续带完整旧历史，而是保留这段 summary，再加上 compaction 之后的新消息继续往下跑
  • prune 会把更早的工具输出清掉，后面发给模型时只保留一个占位文本，而不是原始大段输出

所以，OpenCode 确实有压缩能力，这点没有问题。但它更像“本地会话重写 + 历史裁剪”，不是 provider 原生的 server-side compaction。

这也是我现在最想强调的一点：两边名字一样，不代表应该按同一种收益去理解。

我真正想表达的判断

如果说的是 Codex，那么“compaction 能省 token”这句话我觉得没什么问题，因为 OpenAI 文档已经把 server-side compaction 这件事讲得比较明白了。

但如果说的是 OpenCode，我就不会把话说这么满。更准确的说法应该是：OpenCode 也有 compaction，而且源码能看出来它确实在做会话压缩；但它不是 Codex 文档里那种明确说明的 server-side compaction。

所以我现在更愿意把结论写成一句很直白的话：如果你本来就打算使用 GPT-5.4，那么至少按我现在看到的资料、源码和体感，走 Codex 通常会比走 OpenCode 更省 token。核心不是模型不同，而是 Codex 明确是 server-side compaction，而 OpenCode 更像应用层的本地会话压缩。
